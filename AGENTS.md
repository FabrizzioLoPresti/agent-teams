## Developer Commands

All commands run via `pnpm`. All `db:*` commands load from `.env.local` (via `dotenv-cli`) — **not** `.env`.

```bash
pnpm dev                  # dev server on :3000
pnpm build                # vite build + cp instrument.server.mjs .output/server
pnpm start                # production start (requires prior build)
pnpm test                 # vitest run (non-watch)
pnpm lint                 # eslint
pnpm check                # prettier --write . && eslint --fix

pnpm db:generate          # regenerate Prisma client from schema
pnpm db:migrate           # create + apply migration (prompts for name)
pnpm db:push              # schema push without a migration file
pnpm db:seed              # run tools/scripts/seed.ts
pnpm db:studio            # Prisma Studio GUI
```

Run a single test file: `pnpm vitest run src/path/to/file.test.ts`

## First-Time Setup Order

1. `pnpm install`
2. Copy `.env.example` → `.env.local` and fill in values
3. `docker compose up -d` — starts Postgres 16 + Redis
4. `pnpm db:migrate` — applies migrations and regenerates Prisma client
5. `pnpm dev`

### No manual memoization
**React Compiler is active** (`babel-plugin-react-compiler`). Do NOT add `useMemo`, `useCallback`, or `React.memo`. The compiler handles all memoization automatically.

All server calls go through `src/data/<domain>/<verb>-<entity>.ts` hooks. Never call `client` or `orpc` from `src/orpc/client.ts` directly in a component.

### Code style (Prettier)

## Auth

- Library: Better Auth v1.4
- Three fixed roles set at registration: `user`, `customerComplex`, `ownerComplex`. **Role cannot change after creation.**
- Role-based redirects: `customerComplex` → `/search`, `ownerComplex` → `/dashboard`, `user` → `/`
- oRPC middleware:
  - Public procedures: `baseInputValidationMiddleware`
  - Protected procedures: `authorizedMiddleware` (adds `context.session` and `context.user`, throws `UNAUTHORIZED` if no session)
- Route-level guard: `src/middlewares/auth.ts` — redirects to `/auth/login`
- Session storage: Redis. Session TTL: 7 days.
- Social login: Google only (Apple fields in `.env.example` are not implemented).

## Database

- PostgreSQL 16 via Docker; Prisma v7 with `@prisma/adapter-pg` (explicit driver adapter pattern).
- Schema: `prisma/schema.prisma`. Output: `generated/prisma/` (non-default path — do not change).
- The datasource block has **no `url` field**. Connection string is passed at runtime via `PrismaPg` adapter using `env.DATABASE_URL`. CLI commands get it from `.env.local` via `dotenv -e .env.local --`.
- Soft-deletes only for `Complex` and `Field` (`isActive = false` + `deletedAt`). **Never hard-delete these entities.**
- Timestamps stored as UTC (`@db.Timestamptz`). Open/close hours stored as `HH:MM` strings interpreted in `Complex.timezone` (default: `America/Argentina/Buenos_Aires`).

### Key business rules

- A `FULL` field with `isDividable = true` auto-creates `HALF_A` + `HALF_B` sub-fields on creation.
- When a `Field` is created, `FieldWorkingSchedule` rows are **copied from the parent Complex's `ComplexWorkingSchedule`** — complex must have schedules configured first.
- Cannot soft-delete a `Field` with future `PENDING` or `CONFIRMED` bookings.
- `Booking.version` is used for optimistic locking.

## OpenCode / SpecKit Agent System

Defined in `opencode.json`. The primary agent (`speckit-orchestrator`) **delegates only** — it never writes files. Subagents do the actual work.

Agent prompt files: `.opencode/agent/speckit-*.md`
SpecKit workflow templates: `.specify/`

### Orchestration Rules

1. **Delegate everything.** The Orchestrator never writes, edits, or reads source files. All file operations happen inside subagents.
2. **Decompose first.** Break the feature into phases: spec → clarify → design → plan → consistency check → implement → validate. Each phase maps to exactly one subagent.
3. **Gate on success.** Do not advance to the next phase until the current subagent reports success. On failure, re-delegate with corrective context.
4. **No parallel writes.** Subagents that write files (`speckit-requirements-analyst`, `speckit-architecture-designer`, `speckit-task-planner`, `speckit-implementer`) must never run concurrently against the same feature directory.
5. **Read-only agents may parallelize.** `speckit-consistency-analyzer` and `speckit-review-validator` can run concurrently with each other.
6. **No git operations.** Agents must NOT run `git add`, `git commit`, `git push`, or any other git write command. All version control is managed by the developer.
