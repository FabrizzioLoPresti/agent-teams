# Runbook: Database Migrations with Prisma

Step-by-step guide for applying Prisma schema changes in development and production.

---

## General Flow

```
1. Modify prisma/schema.prisma
2. Create the migration (generates SQL + saves the diff)
3. Apply the migration to the database
4. Regenerate the Prisma client
5. Update TypeScript code if necessary
```

---

## Step by Step

### 1. Modify the Schema

Edit `prisma/schema.prisma` with the desired change. Common examples:

```prisma
// Add a new field
model Complex {
  phone String? @db.VarChar(20)  // ← new
}

// Change a field type
model Booking {
  notes String @db.Text  // was String? @db.VarChar(500)
}

// Add a new model
model Coupon {
  id        String   @id @default(cuid())
  code      String   @unique
  discount  Decimal  @db.Decimal(5, 2)
  createdAt DateTime @default(now())

  @@map("coupon")
}
```

### 2. Create the Migration

```bash
pnpm db:migrate
```

Prisma detects the diff between the current schema and the database state, generates the corresponding SQL, and prompts for a migration name:

```
? Enter a name for the new migration: › add_phone_to_complex
```

**Naming conventions:**
- Use `snake_case`
- Describe the change, not the table: `add_phone_to_complex`, not `update_complex`
- If there are multiple changes: `add_coupon_model_and_booking_notes`

This creates a directory in `prisma/migrations/`:

```
prisma/migrations/
└── 20260322143000_add_phone_to_complex/
    └── migration.sql
```

### 3. Verify the Generated SQL

Before continuing, review the generated `migration.sql` file:

```bash
cat prisma/migrations/20260322143000_add_phone_to_complex/migration.sql
```

Confirm that:
- The SQL does exactly what is expected
- There are no accidental `DROP COLUMN` or `DROP TABLE` statements
- New columns have the correct type
- Indexes and constraints are correct

### 4. Regenerate the Prisma Client

```bash
pnpm db:generate
```

This regenerates `generated/prisma/client` with the updated types. **Always run after any schema change**, even if `db:migrate` was already executed (which does it implicitly).

### 5. Update TypeScript Code

If a new field was added or a type was changed, the TypeScript compiler will detect the errors. Update the affected handlers, Zod schemas, and components.

```bash
pnpm build  # verify there are no type errors
```

## Common Cases

### Adding an optional field

```prisma
model Field {
  notes String? @db.Text  // optional → does not break existing records
}
```

```bash
pnpm db:migrate  # name: add_notes_to_field
pnpm db:generate
```

No risk: existing records will have `notes = NULL` automatically.

### Adding a required field without a default

```prisma
model Complex {
  slug String @unique @db.VarChar(100)  // required, no default
}
```

This **breaks** if there are existing records — PostgreSQL cannot set a value for `slug` on already-existing complexes.

Solutions:

**Option A — Add as optional, migrate data, then make required:**
```prisma
// Step 1: add as optional
slug String? @unique @db.VarChar(100)
```
```bash
pnpm db:migrate  # name: add_slug_to_complex_nullable
```
Then fill in the data (via Prisma Studio or a script):
```bash
pnpm db:studio  # edit manually
```
```prisma
// Step 2: make it required
slug String @unique @db.VarChar(100)
```
```bash
pnpm db:migrate  # name: make_complex_slug_required
```

**Option B — Add with a temporary default in the migration:**
```prisma
slug String @unique @db.VarChar(100) @default("")
```
Set the real values via script, then remove the `@default`.

### Renaming a field

Prisma cannot infer renames — it interprets it as "drop the old field + add new field", which implies **data loss**.

```prisma
// BEFORE
model Complex { description String }

// AFTER — Prisma sees it as drop + add, not as rename
model Complex { summary String }
```

The solution is to manually edit the generated SQL before applying it:

```bash
pnpm db:migrate --create-only  # generates the SQL but does NOT apply it
```

Edit `prisma/migrations/xxx/migration.sql`:
```sql
-- SQL generated by Prisma (INCORRECT):
-- ALTER TABLE "complex" DROP COLUMN "description";
-- ALTER TABLE "complex" ADD COLUMN "summary" TEXT NOT NULL;

-- Correct SQL:
ALTER TABLE "complex" RENAME COLUMN "description" TO "summary";
```

Then apply manually:
```bash
pnpm db:migrate
```

### Removing a field

```prisma
model Booking {
  // specialRequests String? @db.Text  ← remove this line
}
```

```bash
pnpm db:migrate  # name: remove_special_requests_from_booking
```

Verify the generated SQL: it should be `ALTER TABLE "booking" DROP COLUMN "special_requests";`.

> ⚠️ Irreversible in production without a backup. Always back up before running drops.

### Adding an index

```prisma
model Booking {
  @@index([userId, createdAt])  // new index
}
```

```bash
pnpm db:migrate  # name: add_booking_user_created_index
```

No risk of data loss. On large tables, PostgreSQL creates the index without blocking reads (`CONCURRENTLY` — Prisma handles this automatically).

## Reference Commands

| Command | When to use |
|---------|-------------|
| `pnpm db:migrate` | Create and apply the migration (normal flow) |
| `pnpm db:migrate --create-only` | Generate the SQL without applying it (to review or edit) |
| `pnpm db:generate` | Regenerate the Prisma client without touching the DB |
| `pnpm db:push` | Apply the schema directly without creating a migration (dev only) |
| `pnpm db:studio` | Open Prisma Studio to inspect/edit data |

---

## Difference Between `db:migrate` and `db:push`

| | `db:migrate` | `db:push` |
|-|--------------|-----------|
| Creates a migration file | Yes | No |
| Saves history in git | Yes | No |
| Safe for production | Yes | **No** |
| Speed in development | Slower (generates SQL) | Fast |
| When to use | Always for real features | Quick prototyping on an experimental branch |

**Rule:** use `db:push` only to explore changes on a local throwaway branch. Any change going to main must have its migration with `db:migrate`.

---

## Before Merging to Main

Checklist before merging a branch that modifies the schema:

- [ ] The migration is in `prisma/migrations/` and committed
- [ ] The migration SQL was reviewed (no accidental drops)
- [ ] `pnpm db:generate` was run and the client is up to date
- [ ] `pnpm build` passes with no TypeScript errors
- [ ] If seed data is affected, update `prisma/seed.ts`

---

## Rollback

Prisma has no automatic rollback. To revert an applied migration:

**In development:**
```bash
# Option 1: Full reset (destroys all data)
pnpm dlx prisma migrate reset

# Option 2: Write an inverse migration manually
pnpm db:migrate  # name: revert_add_phone_to_complex
# Edit the SQL to undo the previous change
```

**In production:**
Create a new migration that reverts the change (never delete migrations already applied in production). Example:

```sql
-- revert_add_phone_to_complex/migration.sql
ALTER TABLE "complex" DROP COLUMN IF EXISTS "phone";
```

> The golden rule: migrations are append-only in production. Never delete or edit already-applied migrations.
